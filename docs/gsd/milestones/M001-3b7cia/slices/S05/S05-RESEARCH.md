# S05 — Research

**Date:** 2026-04-30

## Summary

S05 is a documentation-and-review slice with two concerns: (1) output consistency across all 9 template types and (2) updating stale help documentation to reflect the post-refactor architecture. The codebase is clean (46 rspec, 6 cucumber all green from S04), so this is purely about content — no Ruby changes expected.

**Output consistency:** 6 of 9 templates use `{{common.metadata}}` inside an HTML comment wrapper (`explanation`, `howto`, `tutorial`, `pr`, `handover`, `fivewhyanalysis`). Three templates deviate: `note.md` has no HTML comment wrapper and no `{{common.metadata}}`; `project.md` has its own GTD-specific metadata in an HTML comment but does not use `{{common.metadata}}`; `adr.md` has no metadata at all (standard ADR format, intentionally minimal). The reference formatting across templates is mostly consistent but uses minor wording variations ("References" vs "Related Resources" vs "Related Topics"). These deviations need collaborative review to determine which are intentional (ADR's minimalism is by design) and which should be normalized.

**Documentation staleness:** `how_to_add_a_new_document_template.md` is **completely stale** — it describes the old 8-step process (create a class file, implement DocumentInterface, update command_router, update command_handlers, update diataxis.rb, update usage_display.rb). The new process is: add a `register` call in `document_types.rb` + create a `.md` template file in `templates/<category>/`. This doc needs a complete rewrite. `how_to_manually_test_all_diataxis_features.md` and `README.md` only mention 4 of 9 document types and need expanding.

## Recommendation

Break this into three tasks: (1) collaborative review of template output consistency with the user, focusing on whether `note.md`, `project.md`, and `adr.md` should adopt `{{common.metadata}}`; (2) complete rewrite of `how_to_add_a_new_document_template.md`; (3) update `how_to_manually_test_all_diataxis_features.md` and `README.md` to cover all 9 document types. An ADR-0015 documenting the registry DSL + template method pattern should also be created since ADR-0008 (class files) and ADR-0012 (external templates) are the most recent related records and the architecture has fundamentally changed since those were written.

## Implementation Landscape

### Key Files

- `docs/how_to_add_a_new_document_template.md` — **complete rewrite needed**. Currently describes: create class file → implement DocumentInterface → update command_router → update command_handlers → update diataxis.rb → update usage_display.rb → add tests → test manually. New process: (1) add `register` call in `lib/diataxis/document_types.rb`, (2) create `templates/<category>/<name>.md` with optional `{{common.metadata}}`, (3) add config key to `Config::DEFAULT_CONFIG` if custom directory needed, (4) run tests. That's it — `UsageDisplay` auto-generates help from registry, `CommandRouter` auto-routes via `DocumentRegistry.lookup`, `CommandHandlers.handle_document` is generic.

- `docs/how_to_manually_test_all_diataxis_features.md` — **expand to cover all 9 types**. Currently only tests howto, tutorial, explanation, ADR. Missing: note, handover, 5why, project, pr. The test structure (create → verify file → verify README → title change → update → verify rename) is sound and should be extended. Also missing: the `.diataxis` config's `projects` key override and the `five_why_analyses`/`handovers`/`notes` config keys.

- `README.md` — **update Usage section**. Currently shows examples for howto, tutorial, explanation, adr only. Add: `dia note new "Title"`, `dia handover new "Title"`, `dia 5why new "Title"`, `dia project new "Title"`, `dia pr new "Title"`. The Appendix section with How-To Guides/Explanations/Projects/Design Decisions is auto-generated by `dia update .` so should be correct, but review after doc updates.

- `templates/references/note.md` — **lacks `{{common.metadata}}`**. Has no HTML comment wrapper at all. Review with user whether to add one (consistency argument) or leave it (note is intentionally lightweight).

- `templates/references/fivewhyanalysis.md` — **has partial metadata** in HTML comment but doesn't include `{{common.metadata}}`. Wait — actually re-reading, it does NOT have an HTML comment wrapper at all. The template body just has raw content. Needs review.

- `templates/references/project.md` — **has its own GTD-specific HTML comment metadata** that doesn't use `{{common.metadata}}`. The GTD philosophy section is unique. Review whether to add `{{common.metadata}}` alongside or leave as-is.

- `templates/references/adr.md` — **intentionally minimal** (standard ADR format). No metadata comment. Likely should stay as-is since ADR format is a community standard.

- `docs/adr/` — **ADR-0015 warranted**. ADR-0008 described the class-file refactoring. ADR-0012 described external templates. Neither documents the registry DSL replacing class files or the template method pattern for custom behavior. A successor ADR should document: (1) registry DSL in `document_types.rb` supersedes per-type class files, (2) template method hooks in `Document` base class, (3) only ADR and HowTo retain custom handler classes.

- `lib/diataxis/document_types.rb` — the registry DSL configuration (9 types registered). Source of truth for how to add a new type. The how-to doc rewrite should reference this directly.

- `lib/diataxis/cli/usage_display.rb` — auto-generates help from `DocumentRegistry.command_names`. The how-to doc should explain this (no manual help text updates needed).

- `lib/diataxis/cli/command_router.rb` — auto-routes document commands via `DocumentRegistry.lookup`. The how-to doc should explain this.

- `lib/diataxis/cli/command_handlers.rb` — `handle_document` is generic for all registered types. No per-type handlers needed for simple types.

## Constraints

- ADR format (adr.md template) is a community standard — don't add metadata comments to it
- `common.metadata` is deliberately narrow (~8 lines of style rules) per D004 — don't expand its scope
- The `project.md` template has GTD-specific philosophy content that serves a different purpose than code-formatting metadata — the two shouldn't be conflated
- The project document type has a known double-prefix filename bug (MEM011) — don't try to fix it here, just document it if relevant

## Common Pitfalls

- **Don't rewrite templates without user approval** — S05 is explicitly a "collaborative review" slice. Template changes (if any) need user sign-off since they affect every future document generated.
- **Don't forget the auto-generated help text** — `UsageDisplay.build_usage_text` dynamically builds command help from `DocumentRegistry.command_names`. The how-to doc must explain this so new contributors don't manually edit help text that doesn't exist as a static string.
- **ADR-0008 status** — Currently says "Proposed". After writing ADR-0015 that supersedes the class-file approach, consider whether ADR-0008's status should be updated to "Superseded by ADR-0015".

## Open Risks

- The user may decide some templates intentionally lack `{{common.metadata}}` and no template changes are needed — in that case T01 resolves quickly and the slice is primarily documentation work.
- The `how_to_manually_test_all_diataxis_features.md` currently has a nested bash block syntax error (Test 6 has a double ```` ```bash ````  opening) — this should be fixed during the update.
